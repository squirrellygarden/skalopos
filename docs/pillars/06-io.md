# Pillar 6 — I/O

## Goals

- One I/O concept in v1; one additional concept in v2; never a zoo.
- Same `(Status, Value)` shape whether the result is returned synchronously or delivered later via a channel message.
- No readiness/polling model. epoll, select, kqueue, and `O_NONBLOCK` are not in Skalapos.
- Cancellation is a first-class operation, not a hack on top.

## Contract — v1 (synchronous blocking)

Every I/O syscall blocks the calling thread until the operation completes:

```c
status_t file_read (handle_t, void*       buf, size_t n, int64_t off, size_t* out_n);
status_t file_write(handle_t, const void* buf, size_t n, int64_t off, size_t* out_n);
status_t chnl_send (handle_t, ...);     // already in pillar 4
status_t chnl_recv (handle_t, ...);
status_t timer_wait(handle_t timer_h);  // blocks until timer fires
```

Concurrency in v1 is **threads**. A program that wants to handle multiple I/O streams concurrently spawns one thread per stream (or a thread pool). The kernel scheduler does the multiplexing.

There is no `O_NONBLOCK`. There is no `select` or `epoll`. A read that has no data to return blocks; a read that has some data returns it (a short read is permitted and explicit via `*out_n`).

Cancellation in v1: the only way to interrupt a blocked thread is to terminate it via `thread_kill(thread_h)` or by terminating the whole process. This is a deliberate constraint of v1 — the cost of doing better is high enough that we wait for v2 to do it properly.

## Contract — v2 (completion-on-channel async)

For every blocking I/O syscall, an `_async` variant is added that returns immediately and posts a completion message on a caller-nominated channel when done:

```c
typedef uint64_t op_tag_t;       // caller-chosen identifier for this submission

status_t file_read_async(handle_t file_h, void* buf, size_t n, int64_t off,
                         handle_t completion_chnl, op_tag_t tag);
status_t file_write_async(handle_t file_h, const void* buf, size_t n, int64_t off,
                          handle_t completion_chnl, op_tag_t tag);
status_t op_cancel(handle_t completion_chnl, op_tag_t tag);
```

The completion message posted to `completion_chnl`:

```c
typedef struct {
    uint32_t evt_kind;     // EVT_OP_COMPLETION
    op_tag_t tag;
    status_t status;
    int64_t  value;        // bytes transferred, or other op-specific result
} evt_op_completion_t;
```

Submission semantics:

- The submission syscall returns immediately with `STATUS_OK` if the op was successfully *queued*. Failure to queue (bad handle, etc.) returns the relevant status without ever posting a completion.
- The op's result is delivered as a message; the receiver matches it to its in-flight submissions by `tag`.
- The caller's buffer (for read) or buffer-and-data (for write) is **loaned to the kernel** for the duration of the op. Until completion arrives or `op_cancel` succeeds, the caller must not:
  - Touch the buffer's bytes
  - Free or unmap the memory
  - Call any syscall that might unmap the region
- After completion or successful cancel, ownership returns to the caller.

Cancellation:

- `op_cancel(chnl, tag)` requests cancellation of a pending op. It is best-effort: the op may already have completed.
- A canceled op still produces a completion message — with `status == STATUS_CANCELED` if cancellation took effect, or with the real result if the op completed before cancel did. Either way, the caller's buffer ownership is returned.

## Pseudocode — userland event loop in v2

```c
// Set up:
handle_t comp;
chnl_create(64, 0, &comp);

// Submit reads from two streams:
file_read_async(file1, buf1, sizeof buf1, -1, comp, /*tag*/ 1);
file_read_async(file2, buf2, sizeof buf2, -1, comp, /*tag*/ 2);

// Event loop:
for (;;) {
    evt_op_completion_t m;
    size_t mlen;
    chnl_recv(comp, &m, sizeof m, &mlen, NULL, 0, NULL);

    switch (m.tag) {
        case 1: handle_stream1(buf1, m.status, m.value); break;
        case 2: handle_stream2(buf2, m.status, m.value); break;
    }
}
```

No `O_NONBLOCK`, no readiness, no level-vs-edge triggering. The same channel that delivers lifecycle events can deliver I/O completions; a sufficiently advanced loop multiplexes both.

## Why this over alternatives

- **Readiness model (epoll/kqueue/select)** — exists in Unix because non-blocking I/O came before real async I/O. Skalapos has no such history. Readiness adds an extra syscall per op (subscribe + wait + read), suffers spurious wakeups, has the edge-vs-level distinction, and *doesn't help with disk I/O* (a "ready" disk is a contradiction). Skipped entirely.
- **Build async into v1** — considered. Sync variants would be libc wrappers (submit + chan_recv). Rejected because it puts pressure on the channel/handle/spawn implementation before any of them are battle-tested. Sync-first gives us a working OS sooner; async is an additive extension, not a redesign.
- **Sync-only forever** — workable for a toy OS, but rules out interesting userland (TCP servers, multiplexed terminals). Worth keeping the door open via the v2 plan even if we never walk through it.
- **POSIX AIO** — exists; nobody uses it. Skalapos's async model is io_uring-shaped without the ring buffers (we deliver via channels, not a shared SQ/CQ memory ring). Rings are a perf optimization layered on top of message-based completion; not v1, not v2.

## v2+ direction

- **Submission rings.** Once `_async` exists, add an opt-in shared-memory submission ring (one syscall to register, then submit by writing to memory). Pure io_uring with completion via channel rather than CQ. Optional perf path.
- **Linked submissions.** "Do A, and if it succeeds, do B with A's result as input." Saves round-trips for chains of ops.
- **Vectored I/O.** `file_read_v(handle, iov[], n)` for scatter/gather. Useful once anything cares about perf.
